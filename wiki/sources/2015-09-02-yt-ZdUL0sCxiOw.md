# UX Design 2: How To Design a Website: User Stories

## Summary

Part two of a live UX walkthrough on The Futur, filmed at Chris Do's machine but
**led methodologically by Jose Caballer**. Jose demonstrates two connected exercises:
(1) a **goals prioritization** matrix scored on "obtainability" vs. "desirability" (1–10
each), and (2) building a **user story / customer journey narrative** for a website
redesign, walked through a real client — "Oles," a small fishing resort — using a
customer profile ("Joe," a returning executive/CFO angler).

Jose owns the framework, the scoring rules, the journey structure (find out → engage →
return → refer), and the strategic guardrails (viral loop, feature matrix, discovery
phase, budget-driven scope). **Chris's role is domain-expert and facilitator**: he
supplies the real-customer knowledge (he is himself a customer of the resort, so Jose
repeatedly turns to him for authentic detail), asks framing questions, and — at the
close — reflects on the value of the method. Andrew and Jamie (crew) ask two audience
questions near the end.

Because the substantive design methodology is Jose's, most content here is **CONTEXT**,
not Chris-persona material. Chris-attributed statements are limited to facilitation, the
real-customer color, and one closing reflection. Attribution marked `uncertain` for any
line where the speaker is ambiguous.

## Key claims

_(All dated 2015-09-02; paraphrased unless quoted below. Speaker attributed per claim.)_

**Jose Caballer (methodology — CONTEXT):**
- Run a goals-prioritization exercise: list the client's marketing/business goals, then
  score each on **obtainability** (how easy/low-effort, 1–10) and **desirability** (how
  badly they want it, 1–10). High-desirability + high-obtainability items rise to the top
  of the to-do list. Re-run the same exercise scoped specifically to the website.
- The exercise forces alignment because "everybody might have different ideas" about what
  content is a priority — including the designer, who otherwise is "left flailing in the
  wind."
- A **user story** (a.k.a. customer story) builds "the narrative of what a user did in
  time," so you can meet that user with the features they need to accomplish their goal.
- Structure each journey around a customer profile and four beats: how they **find out**,
  how they **engage**, how they **return**, and how they **refer**.
- Build the story live with the client (e.g., a shared Google Doc over a screen-share),
  pulling one customer profile at a time and reasoning about behavior in context of the
  site.
- When new feature ideas surface (group trip planning, shared availability calendar),
  **note them, then check competitors** and how deep they go into booking — because the
  real question is "how much of that effort cost-wise is going to return actually new
  customers."
- Missing stakeholder in the room = the **developer**; novel creative ideas can quietly
  become "a $30,000 build" — "put a pin on that as an idea."
- A referral mechanic (invite buddies → they invite more) is a **"viral loop."**
- Method for avoiding feature bloat: run a **discovery phase** to establish the client's
  technology goals and budget first; then either constrain brainstorming up front, or go
  "sky's the limit," scope it out, and cut to fit budget.
- Take all features generated and run a **feature matrix**, prioritizing by cost,
  complexity, and necessity.
- Web design is "completely nonlinear" — a website is a nonlinear experience; over-
  controlling it makes it too difficult ("you have to flow like Bruce Lee").
- User stories feed the next deliverables: **site map → wireframes** (covered next episode).

**Chris Do (facilitation + real-customer context — mostly CONTEXT):**
- Supplies authentic customer detail for "Joe": found the resort as a guest brought by a
  friend, signs up for the newsletter automatically because everyone who books gives their
  email, has an upcoming trip and hasn't decided yet. (Chris is a real customer of the
  resort, per Jose's prompt.)
- Domain facts about the business: trips cap at **24 people**, roughly **$3,000/person**,
  a group of ~5 is a ~$15,000 investment — so a highly personalized, high-touch service,
  which argues **against** requiring full online booking/payment.
- Suggests the invitation to friends should feel like a **postcard** ("paperless post"),
  because you have to sell the friends/spouses/cousins on going.
- Reframes a feature idea: a Facebook "who wants to go fishing with me" post could satisfy
  the client's separate goal of being more present on Facebook/Twitter — "I don't think
  anybody's even doing that in this space."
- Suggests scaffolding the booking flow: plan → **hold/reserve** → book (don't jump
  straight from plan to book).

**Andrew / Jamie (crew, audience Qs — CONTEXT):**
- How do you resolve conflicts between different user stories? (Jose: build all the user
  types' stories; a website is nonlinear, not everything is available to everyone.)
- How do you avoid feature bloat when simplifying a complex site? (Jose: discovery phase +
  budget constraint, answered above.)
