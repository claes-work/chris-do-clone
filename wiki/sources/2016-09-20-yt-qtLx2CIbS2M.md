# Graphic Design Tutorial: Web Design Review pt. 2

★ L3-candidate — adds a distinct **designer→developer handoff** principle (annotate, don't edit the code) and a **first-pass review posture** that go beyond generic design-craft.

## Summary

Episode of "The Process" (2016 era). Chris Do and co-host **Jose Caballer** (context) review the
developers' first coded pass of the redesigned **Blind** website (Blind = Chris's design studio, context).
Jose narrates each page as it loads live (blog index, sample article, capabilities, events, team, case
studies, all-projects index, work detail); Chris reacts with design notes and — more instructively — coaches
Jose on *how a designer who is leading a project should communicate changes back to developers*. The
recurring teaching beat: don't reach into the code yourself; screen-capture, annotate, and let the developer
own the CSS. Chris also restates *why* the site is being rebuilt: the studio's messaging changed (leading
with brand strategy instead of motion design), plus content/standards go stale over time.

Attribution note: Chris Do's critique lens and process coaching = **persona**. Jose Caballer's page-by-page
narration and the reviewed work = **context**. Ambient banter (Matt, Dan, "camera debut") is noise, omitted.

## Key claims

Per speaker; paraphrased unless quoted below. All dated 2016-09-20.

**Chris Do (persona — critique / process):**
- A designer leading a project should **not edit the developers' CSS or change values in Firebug themselves**; instead screen-capture the page, annotate it (e.g. a red arrow: "move that up, a little bigger"), and communicate the change so the developer owns the code.
- Framing analogy: it would feel wrong if a *client* opened your design file and started moving things around — same reason you don't move things in the developer's stylesheet.
- The designer has to **let go of control issues** and allow a **margin of error**; changes like "make it a bit smaller, bump it a point or two" should be handled by working *with* the developers, not around them.
- **Web animation should be used selectively** — page titles animating in is "too much"; be deliberate about where animation goes.
- First-pass notes: **animations need to be much faster**; several sections are **too tall (~2/3 of current size)**; some case-study images should not go full-bleed but scale to the window; hover states should shift image color and reveal the project name.
- Correct **review posture on a first pass**: nothing should be "jumping out saying this is the worst thing ever" — a promising first pass gets marked-up notes, not a panic call to the developers.
- **Why rebuild the site (rationale):** (1) content/format goes stale if you walk away and never update it; web standards, screen sizes, and technology keep changing; (2) more importantly, the studio's **messaging changed** — they now lead with brand strategy rather than motion design.
- **Content hierarchy:** talk about *what you're doing*, then show **case studies**, then the work — people should care about the message/intent *before* being shown the work.
- Because it's an internal project, Chris "manages the client" (himself); once launched, the sales team will sell differently, leading with brand strategy — closing the ~2.5-year arc of proving that positioning.

**Jose Caballer (context — presenting the build):**
- Blog index features articles that load on scroll; pagination still needs to be added back.
- Inspiration quotes shouldn't resize per quote — should be a fixed size that accommodates the longest quote (Chris: agrees, "I don't like it bouncing up and down").
- Sample article / capabilities page is "almost spot-on… pretty close to pixel perfect" to the design.
- All-projects index uses **infinite scroll** (loads more at the bottom, no clicking) and three-column layout; case studies use full-width images; a third "featured" index is the last variant.
- Work-detail page mirrors the case study but with less layout variation as you go deeper (two columns at most), credits moved to the bottom instead of a top tab.
- Development was scheduled to finish by end of September 2016; next review will look at the site functioning with working links.
